# NISABA — Tech Stack

## Decision Summary

| Decision | Choice | Reasoning |
|---|---|---|
| Storage format | SQLite | Single file, zero config, zero maintenance, trivial backup on TrueNAS. Perfectly capable for ~2000 games with one user. |
| Backend language | Go | User has some Go experience, explicit and readable code, single binary deployment, widely used in homelab self-hosted apps. |
| Router | Chi | Lightweight, idiomatic Go HTTP router. Minimal magic. |
| Database layer | sqlc | User knows SQL — write plain SQL queries, sqlc generates type-safe Go. No ORM abstraction. |
| Templates | html/template | Go's built-in HTML templating. Safe, simple, readable. |
| Frontend | HTMX | HTML attributes that make HTTP requests. No framework mental model required. |
| Styling | TailwindCSS | Utility classes directly in HTML. No separate CSS files to manage. |
| Container | Single binary + nginx | Go compiles to one binary. Nginx handles reverse proxy and static files. |
| Deployment | TrueNAS SCALE Apps | Docker containers managed via TrueNAS app store where available. |

## Stack Detail

### Backend — Go

Go compiles the entire application to a single binary with no external runtime
dependencies. Deployment on Atlas is a minimal Docker container:

```
FROM scratch
COPY gamerepo /gamerepo
CMD ["/gamerepo"]
```

Go is explicit by design — no hidden behaviour, no framework magic.
What you read is exactly what runs.

### Routing — Chi

```go
r := chi.NewRouter()
r.Get("/library", handlers.GetLibrary)
r.Post("/library/sync", handlers.SyncLibrary)
r.Get("/wishlist", handlers.GetWishlist)
```

URL → handler. One-to-one, easy to trace.

### Database — SQLite + sqlc

SQLite stores the entire database as a single file on Atlas.
sqlc generates type-safe Go functions from plain SQL files.

Workflow:
1. Define tables in `schema.sql` (standard SQL DDL)
2. Write queries in `queries/*.sql` (standard SQL)
3. Run `sqlc generate` → Go functions appear in `db/` directory
4. Call those functions from handlers

The SQL files are always readable and editable independently of the Go code.

### Frontend — HTMX

HTMX extends HTML with attributes for making HTTP requests:

```html
<!-- Clicking this button sends GET /library/sync -->
<!-- and puts the response HTML into #sync-status -->
<button hx-get="/library/sync"
        hx-target="#sync-status"
        hx-swap="innerHTML">
  Sync All
</button>
```

The server returns HTML fragments, not JSON. The mental model:
- User action → HTTP request → server returns HTML → HTMX drops it into the page
- No JavaScript framework, no component lifecycle, no state management

### Styling — TailwindCSS

Utility classes applied directly in HTML templates:

```html
<div class="flex items-center gap-4 p-4 bg-gray-900 rounded-lg">
  <img class="w-16 h-16 rounded" src="{{ .Artwork.Cover }}" />
  <h2 class="text-xl font-bold text-white">{{ .Title }}</h2>
</div>
```

No separate CSS files to maintain. Styles live next to the HTML they affect.

## Deployment on Atlas (TrueNAS SCALE)

Two containers managed via docker-compose:

```
┌─────────────────────────────────┐
│  nginx (reverse proxy)          │
│  Port 80/443 → :8080            │
└────────────────┬────────────────┘
                 │
┌────────────────▼────────────────┐
│  gamerepo (Go binary)           │
│  Port :8080                     │
│  Volume: /data/gamerepo.db      │  ← SQLite file on Atlas storage
└─────────────────────────────────┘
```

The SQLite file lives on a TrueNAS dataset, snapshotted automatically by ZFS.
Backup = ZFS snapshot. No additional backup strategy needed.

## Schema Translation

The TypeScript schemas in this directory translate to Go structs and SQL tables:

| TypeScript | Go equivalent | SQL equivalent |
|---|---|---|
| `interface GameRecord` | `type GameRecord struct` | `CREATE TABLE games` |
| `string` | `string` | `TEXT` |
| `number` | `int64` / `float64` | `INTEGER` / `REAL` |
| `boolean` | `bool` | `INTEGER` (0/1) |
| `string \| null` | `sql.NullString` | `TEXT NULL` |
| `string[]` | `[]string` (JSON column) | `TEXT` (JSON) |
| nested objects | JSON column or joined table | `TEXT` (JSON) or separate table |
